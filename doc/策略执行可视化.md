# 策略执行可视化与全链路审计计划

## Summary

目标是把策略从“只在发信号时可见”改成“每一步判断都可追踪”。第一版采用 **Bar 全量审计 + 关键 Tick 审计**，并在图表右侧新增 **策略页**，实时展示当前策略走到哪一步、每个条件是否满足、差多少触发，以及历史审计流。

## Key Changes

- 新增策略过程事件模型 `StrategyTraceRecord`：
  - 字段包含 `trace_id`、`instance_id`、`strategy_id`、`symbol`、`timeframe`、`mode`、`event_type`、`event_time`、`step_key`、`step_label`、`step_index`、`step_total`、`status`、`reason`、`checks`、`metrics`、`signal_preview`、`created_at`。
  - `event_type` 取值：`bar`、`key_tick`、`signal`、`order_plan`、`order_result`。
  - `checks` 为数组，每项包含条件名、是否满足、当前值、目标值、差值、说明。
- Python 策略服务新增统一返回字段：
  - 对无信号判断也返回 `trace`，不再只返回 `_no_signal`。
  - `ma20.pullback_short` 第一版输出明确步骤：
    1. 等待 MA20 数据足够
    2. 等待跌破 MA20
    3. 已跌破，等待反抽触碰 MA20
    4. 等待跌破触碰K开盘价
    5. 已触发/已完成
  - Tick 只在进入关键等待状态后记录，即 `WAIT_BREAK_TOUCH_OPEN` 后记录关键 Tick。
- Go 策略管理器新增审计链路：
  - `callDecision` 接收 Python 返回的 `trace`。
  - Bar trace 全部落库并广播。
  - Tick trace 仅在 `trace.event_type=key_tick` 时落库并广播。
  - 发出信号时同时写 `strategy_signals`、`strategy_traces`、`order_audit_logs`，保证信号前后上下文连续。
- 新增 API 和 WebSocket：
  - `GET /api/strategy/traces?instance_id=&symbol=&limit=`
  - WebSocket 事件：`strategy_trace_update`
  - 保留现有 `/api/strategy/signals` 和 `/api/orders/audit`，不破坏旧页面。
- 图表右侧新增 `策略` tab：
  - 顶部显示当前运行实例、连接状态、最近事件时间、当前步骤 `step_index/step_total`。
  - 中部显示步骤进度条和条件 checklist。
  - 下方显示审计流：Bar 判断、关键 Tick、信号、订单计划、订单结果。
  - 点击审计流中的事件可定位到对应 K 线时间；若该时间不在当前窗口，先保留为可点击但不强制加载历史窗口。

## Data Flow

- 行情进入 Go `HandleRealtimeBar/HandleReplayBar/HandleRealtimeTick/HandleReplayTick`。
- Go 调 Python `OnBar/OnTick/OnReplayBar`。
- Python 每次策略判断返回：
  - `no_signal=true/false`
  - `reason`
  - `metrics`
  - `trace`
- Go 写入 `strategy_traces` 并通过 WebSocket 广播。
- 前端 `TradingTerminal.vue` 接收 `strategy_trace_update`，更新右侧策略页。
- 信号产生后继续走现有 `persistDecision` 和订单审计逻辑，同时把订单计划/结果追加为 trace。

## Test Plan

- Go 单元测试：
  - 无信号 Bar 判断会写入 trace，但不写入 signal。
  - 关键 Tick 判断会写入 trace。
  - 非关键 Tick 不写 trace。
  - 信号产生时 trace、signal、order audit 都存在且 instance/symbol/event_time 对齐。
- Python 测试：
  - `ma20.pullback_short` 每个状态都返回正确 `step_key`、`step_index`、`checks`。
  - 等待 MA20、跌破 MA20、反抽触碰、关键 Tick 触发四类场景覆盖。
- Web/API 测试：
  - `GET /api/strategy/traces` 支持按 instance 和 symbol 过滤。
  - WebSocket 收到 `strategy_trace_update` 后图表页策略 tab 更新最新步骤。
- 构建验证：
  - `go test ./internal/strategy ./internal/web`
  - `npm --prefix web run build`

## Assumptions

- 第一版只重点适配 `ma20.pullback_short`，但 trace 数据结构设计为通用策略可复用。
- 审计粒度按用户选择：Bar 全量、关键 Tick 记录，不做全部 Tick 落库。
- 审计数据持久化到数据库，后续需要再补保留天数或清理策略；第一版不自动删除。
- 策略可视化主入口放在图表右侧新增 `策略` tab，不放到首页策略管理区。
