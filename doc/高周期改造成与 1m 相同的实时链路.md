# 高周期改造成与 1m 相同的实时链路

## Summary
把 `5m/15m/30m/1h/1d` 从“图表层拿 `history1m + currentPartial` 临时重算”改成“和 `1m` 一样由实时状态机维护当前桶、每 tick 更新 partial、跨桶时发 final、final 才落库”的链路。

本次改造的落地口径固定为：
- `1d` 按交易日，不按自然日。
- `5m/15m/30m/1h/1d` 的分桶边界继续按交易时段分钟序列确定，和现有历史/mm 查询保持一致。
- 盘前盘后、非交易时段 tick 继续按 `1m` 现有规则处理，不生成 bar。
- 对外接口不改：REST `/api/kline/bars`、WS `chart_bar_update`、timeframe 文本值都保持不变。

## Key Changes
### 1. 抽出统一的“实时分桶规划器”
在聚合层新增一套可同时供历史聚合和实时状态机复用的 bucket planner，输入为：
- 交易时段 minute map
- timeframe (`5m/15m/30m/1h/1d`)
- 当前 bar 的 `DataTime/AdjustedTime`
- 当前交易日 key

输出固定为：
- bucket 唯一键
- bucket 标签时间 `DataTime`
- bucket 标签时间 `AdjustedTime`
- 该桶是否跨 session
- 该桶所属 trading day key

规则固定如下：
- `5m/15m` 按 session 内分钟序列切桶。
- `30m/1h` 保持现有口径，按交易分钟序列切桶，不改成自然整点。
- `1d` 按交易日聚合，夜盘 + 次日日盘属于同一 trading day。
- `1d` 标签与历史重建统一为“该交易日首个交易分钟标签”，`DataTime` 用交易日日期，`AdjustedTime` 用实际开盘日期。

这套 planner 需要同时被批量聚合路径和实时路径使用，避免历史与实时再次分叉。

### 2. 合约高周期改成 runtime 自维护状态
在 `marketDataRuntime` 的 per-instrument state 中新增 `5m/15m/30m/1h/1d` 的当前桶状态，状态内容与 `1m` 同级别维护：
- 当前 bucket key
- 当前 partial bar
- 当前桶开桶时的累计成交量基线
- 当前 trading day key
- 最近一次已发 final 的 bucket key/时间

每个有效 tick 到来时：
- 先继续按现有逻辑更新 `1m`。
- 再对 `5m/15m/30m/1h/1d` 逐个判断该 tick 所属 bucket。
- 若仍在当前 bucket：
  - 直接增量更新 `Open/High/Low/Close/Volume/OpenInterest/SettlementPrice`
  - 立即发该周期 `partial`
- 若进入新 bucket：
  - 先把上一桶作为 `final` 发出并生成 persist task
  - 再用当前 tick 初始化新桶，并立即发新桶 `partial`

成交量计算规则与 `1m` 完全一致：
- 用该周期桶起点的累计成交量做基线
- 当前桶 `Volume = 当前 tick 累计量 - 桶基线`
- 不再通过重扫 `history1m` 反推 volume

高周期 final bar 的持久化仍写入现有 mm 表：
- 合约写 `future_kline_instrument_mm_*`
- schema、主键、查询接口不改
- DB 中只存 final，不存 partial

### 3. L9 高周期也改成状态机，不再临时重算
L9 当前 `1m partial` 仍保持现有来源：按最新 constituent tick 生成 L9 当前分钟 partial。  
在此基础上，给 L9 root 增加与合约相同的高周期状态机：
- 输入不是“重扫 `history1m`”，而是“每次 L9 1m partial/final 事件”
- `5m/15m/30m/1h/1d` 通过增量状态推进
- final 仍落到现有 `future_kline_l9_mm_*` 表
- 图表层不再根据 `history1m + currentPartial` 临时拼 `5m+`

这意味着 chart realtime 对 contract 和 L9 都统一成“收到某周期 bar 事件就覆盖最后一根”的简单模式。

### 4. ChartStream 改成缓存各周期最新状态，不再负责高周期计算
`ChartStream` 的职责收缩为“接收 bar/tick 事件并广播”，不再承担高周期 partial 聚合：
- 保留 quote/tick 缓存逻辑
- `1m`/`5m`/`15m`/`30m`/`1h`/`1d` 都只缓存 runtime/L9 状态机已经算好的最新 partial/final
- `buildPartialBarForTimeframe` 这类基于 `history1m` 的高周期现算逻辑移除或只保留为测试/兜底，不作为主链路
- 订阅时直接返回当前 timeframe 的最新 bar snapshot；若该周期当前没有 partial，则返回最后一根 final；若进程刚启动且尚未恢复当前桶，则只返回历史 final

WS 消息形状不变：
- `ChartBarUpdate.Subscription`
- `phase=partial|final`
- `bar.adjusted_time` 仍是前端唯一覆盖键

### 5. 重启恢复改成“按当前交易日 1m 历史回灌状态”
高周期改成状态机后，不能接受“服务重启后直到下一桶才算准”。恢复策略固定如下：

合约：
- 某合约在启动后第一次收到有效 tick 时，先从对应 `1m` 表读取“当前 trading day 已闭合的 1m final bars”
- 依次喂给 `5m/15m/30m/1h/1d` 状态机恢复当前桶
- 再处理这笔 live tick，立刻得到正确 partial

L9：
- 第一次产生 L9 实时事件时，从对应 `L9 1m` 表读取当前 trading day 已闭合的 `1m final`
- 先恢复高周期状态，再接着处理这笔 L9 live partial/final

恢复边界固定：
- 只回灌当前 trading day，不扫跨交易日全量历史
- 只用 DB 中 final `1m` 恢复；当前未闭合分钟仍由 live tick/live L9 partial 接上
- 若启动后尚无新 tick，则图表只展示 REST 拉到的历史 final，不伪造 partial

## Finalize Rules
高周期 finalize 全部采用“和 1m 一样的跨桶闭合规则”，不引入定时器、不在盘后主动补 synthetic close：

- `1m`
  - 第一笔落入下一分钟的 tick 到来时，上一分钟 final
  - `flush/replay stop` 时补最后一根

- `5m/15m/30m/1h`
  - 第一笔落入下一 bucket 的 tick 到来时，上一桶 final
  - `flush/replay stop` 时补最后一根

- `1d`
  - 第一笔落入下一 trading day 的 tick 到来时，上一交易日 final
  - `flush/replay stop` 时补最后一根
  - 不新增盘后定时 finalize；和 `1m` 一样，靠“下一桶开始”或 `flush` 触发闭合

## Test Plan
1. 合约实时链路
- 单周期用例：`5m` 每 tick 更新 partial，跨桶时只发一次 final，再开新 partial
- 多周期并行用例：同一 tick 能同时推进 `5m/15m/30m/1h/1d`
- volume/open/high/low/close/open_interest 与手工推导一致
- 非交易时段 tick 与 `1m` 一致被丢弃，不推进任何高周期状态

2. 交易时段与标签
- `5m/15m` 在 session 内按分钟序列切桶
- `30m/1h` 保持现有历史标签与 bucket 对齐，不切到自然整点
- `1d` 夜盘 + 次日日盘归到同一 trading day，标签与离线重建一致

3. 历史一致性
- 同一批 `1m` 数据，实时状态机生成的 final bars 与 mm 重建结果逐根一致
- `/api/kline/bars` 加载出的最后一根历史，与 websocket 推来的同周期 final/partial 使用同一 `adjusted_time`
- 前端 `mergeRealtimeBarUpdate` 只覆盖，不重复 append

4. 重启恢复
- 进程中途重启后，第一笔 live tick 到来即可恢复正确的 `5m/15m/30m/1h/1d partial`
- 若启动后尚无新 tick，图表只显示历史 final，不产生错误 partial
- L9 重启后同样可由 `L9 1m final + 首笔 live L9 事件` 恢复当前高周期状态

## Assumptions
- 不改 REST/WS 对外协议，也不新增新 timeframe 字面值。
- mm 表继续只存 final bar；partial 仅通过 websocket 推送。
- contract 和 L9 都采用相同的“状态机实时链路”目标，不保留 chart 层高周期临时重算作为主逻辑。
- `1d` 的正确业务口径以交易日为准；现有自然日/`00:00` 相关逻辑视为待替换旧实现。
