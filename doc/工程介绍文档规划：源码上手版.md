# 工程介绍文档规划：源码上手版

## Summary

目标是产出一份“让新接手开发者快速掌握工程”的系统化介绍，输出顺序采用“先总纲，再逐章细化”。

整体介绍按以下大维度展开，顺序固定：

1. 工程定位与核心能力
2. 运行形态与系统边界
3. 代码目录与模块分工
4. 核心主链路：实时行情从 CTP 到落库
5. 核心辅链路：回放、导入、查询、策略、交易
6. 存储模型与关键数据表
7. Web/API 与前后端协作方式
8. 配置体系与运行环境
9. 测试、调试与排障抓手
10. 推荐阅读路径与二次开发切入点

## Key Changes

### 1. 先给“大的方面”总览

首段先用高层视角回答 4 个问题：

- 这是一个什么系统
- 它解决什么问题
- 它由哪些子系统组成
- 新人先从哪些部分建立整体心智模型

总纲部分不下沉到函数级别，只介绍模块职责和它们之间的关系，明确这是一个以 `quotes -> replay/importer/klinequery/web -> strategy/trade` 为主轴的期货行情与图表/策略平台。

### 2. 再按总纲逐章细化

每个大章都用统一模板展开，避免介绍风格散：

- 本章回答什么问题
- 这个模块在系统里的位置
- 关键入口/核心对象/核心数据
- 它和上下游如何交互
- 新人最容易误解的点
- 建议先看哪些文件

各章重点如下：

- 工程定位与核心能力：
  用 README 和现有功能边界说明它不是单一采集器，而是“实时行情、回放、导入、图表、策略、交易、配置、状态监控”组合体。
- 运行形态与系统边界：
  说明 live/replay、paper/live trade、shared meta / market realtime / replay data 的边界，避免新人把实时和回放链路混在一起。
- 代码目录与模块分工：
  以 `internal` 下的核心目录为主，解释 `quotes`、`replay`、`web`、`strategy`、`trade`、`importer`、`calendar`、`klinequery`、`searchindex`、`db/config` 的职责，不做逐文件罗列。
- 实时行情主链路：
  重点细化 `OnRtnDepthMarketData -> mdSpi -> marketDataRuntime -> shard -> fileWriter/dbWriter -> klineStore -> status/websocket`，这是全工程最重要的一章。
- 辅链路：
  分别解释：
  - replay 如何把 tick CSV / bus 事件重新送回行情链路
  - importer 如何把通达信历史分钟线导入 MySQL
  - klinequery/searchindex 如何支撑前端图表与检索
  - strategy/trade 如何叠加在行情系统之上
- 存储模型与关键表：
  讲清 1m、mm、L9、用户图表、索引、交易日历等表的命名和用途，不讲建表 SQL 细节。
- Web/API：
  解释前端 Vue 控制台如何通过 HTTP + WebSocket 驱动运行、监控和操作系统。
- 配置与环境：
  重点讲 `config.json` 的几大分区、Windows/CTP DLL 依赖、MySQL、前端、Python 策略服务。
- 测试与排障：
  说明已有测试组织、`testmysql`、关键日志点、运行状态页、tick CSV、队列/延迟指标的价值。
- 阅读路径：
  给出“新人第一天怎么读”的顺序，而不是只给静态知识点。

### 4. 输出风格

文风固定为“源码上手版”，强调：

- 先建立整体结构，再下钻
- 少做概念堆砌，多讲模块关系和数据流
- 多讲“为什么这样设计”，少讲表面字段枚举
- 每章结尾给一个“新人该先看哪里”的建议

## Acceptance / Review Criteria

这份介绍完成后，应满足以下验收标准：

- 一个没接触过本项目的 Go 开发者，读完总纲后能说清工程的 5-8 个核心子系统。
- 读完实时行情主链路章节后，能说出 tick 是如何从 CTP 回调一路走到文件和 MySQL 的。
- 读完辅链路章节后，能区分实时、回放、导入、查询、策略、交易这几条链路的边界。
- 读完阅读路径章节后，能按建议顺序开始看代码，而不是在目录里盲找入口。
- 整体内容以“快速掌握”为目标，不写成逐文件 API 手册，也不写成泛泛产品介绍。

## Assumptions

- 受众固定为“接手源码的开发者”，不是终端用户。
- 输出采用“先总纲，再逐章细化”的形式，而不是一次性无结构长文。
- 介绍重点放在当前代码库实际实现，不补充超出仓库事实的架构设想。
- 默认不加入时序图或 Mermaid 图；如果后续需要，可以在实时主链路章节单独追加。
