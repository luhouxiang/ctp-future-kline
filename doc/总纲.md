**总纲**
这是一个以国内期货 CTP 为实时入口、以 MySQL 为核心存储、以 Web 控制台为操作面的综合型行情平台。它不只是“采 1 分钟线”的工具，而是一套把实时行情、历史导入、回放复现、图表查询、策略接入、模拟交易、状态监控整合在一起的系统。

它主要解决三类问题。第一类是“把交易所实时行情可靠地接进来，并沉淀成结构化数据”；第二类是“把历史与实时统一到同一套查询和图表体系里”；第三类是“在统一数据底座上继续叠加回放、策略、交易和运维能力”。

从系统构成上看，可以把它拆成 7 个核心子系统：`quotes` 实时行情主链路，`replay` 回放调度，`importer` 历史导入，`klinequery/searchindex` 查询与检索，`web` 前后端控制台，`strategy` Python 策略接入，`trade` 交易服务。新人建立整体心智模型时，建议先抓住主轴：`quotes -> replay/importer/klinequery/web -> strategy/trade`。

新人最先要形成的认识是：这个工程的“根”在实时行情链路，其他大部分能力都是围绕这条链路做复用、回灌、查询或控制。先看 [README.md](/e:/work/go/ctp-future-kline/README.md)，再看 [internal/quotes](/e:/work/go/ctp-future-kline/internal/quotes)。

**1. 工程定位与核心能力**
这一章回答：这是个什么工程，它的能力边界在哪里。

它本质上是一个“期货行情与图表/策略平台内核”。实时层接 CTP 行情，聚合成 1m 和更高周期 K 线；存储层落 MySQL；控制层通过 Web 页面驱动运行；扩展层继续叠加回放、策略、交易、日历、导入、检索。

它的核心能力可以概括为：

- 实时接收 CTP tick，并生成 1m、mm、L9
- 把 tick 和 K 线存成可查询、可回放、可复现的数据资产
- 提供图表、搜索、状态监控和 Web 控制能力
- 支持通达信历史数据导入，补齐历史库
- 支持 replay，把历史 tick 再送回实时链路
- 支持 Python 策略服务和模拟/实盘交易能力

最容易误解的点是：它不是单一微服务，而是一个“多子系统同仓工程”；也不是只做分钟线数据库，而是围绕分钟线构建了完整控制面和扩展面。

建议先看 [README.md](/e:/work/go/ctp-future-kline/README.md)。

**2. 运行形态与系统边界**
这一章回答：这个系统有哪些运行模式，各模式的边界是什么。

最重要的边界有两组。

第一组是行情来源边界：

- `live`：真实 CTP 行情进入系统
- `replay`：从 tick CSV 或 bus 日志重放，重新驱动同一条行情处理链路

第二组是交易后端边界：

- `live_trade`：真实交易后端
- `paper_live`：实时行情 + 模拟交易
- `paper_replay`：回放行情 + 模拟交易

再往下是数据边界。代码里把共享元数据、实时市场数据、回放市场数据、图表用户数据、交易数据拆成不同逻辑库，避免把“公共元信息”和“实时/回放业务数据”混在一起。这部分入口主要在 [internal/config/config.go](/e:/work/go/ctp-future-kline/internal/config/config.go)。

最容易误解的点是：回放不是独立的一套行情计算系统，而是通过 replay sink 把历史事件重新送回 `quotes` 主链路，因此实时和回放的行为尽量保持一致。

建议先看 [internal/config/config.go](/e:/work/go/ctp-future-kline/internal/config/config.go) 和 [internal/replay](/e:/work/go/ctp-future-kline/internal/replay)。

**3. 代码目录与模块分工**
这一章回答：代码主要分布在哪里，每个目录负责什么。

核心目录分工如下：

- `internal/quotes`：实时行情主系统，含 CTP 接入、tick 处理、分钟线聚合、L9、DB writer、文件 writer、状态中心
- `internal/replay`：回放任务调度中心，从 tick CSV 或 bus 读取事件并分发
- `internal/importer`：通达信历史数据导入
- `internal/klinequery`：图表查询、bars 聚合、指标输出
- `internal/searchindex`：K 线搜索索引
- `internal/web`：HTTP API、WebSocket、静态页面服务
- `internal/strategy`：Python gRPC 策略服务管理、实例、信号、回测
- `internal/trade`：交易服务和账户/订单/成交查询
- `internal/calendar`：交易日历导入与维护
- `internal/db` / `internal/config`：数据库和配置基础设施
- `web`：Vue 前端控制台

最容易误解的点是：`quotes` 不是单个小模块，而是本工程最大的核心域；`web` 也不是纯展示层，它是系统的控制入口。

建议先看 [internal/quotes](/e:/work/go/ctp-future-kline/internal/quotes)、[internal/replay](/e:/work/go/ctp-future-kline/internal/replay)、[internal/web/server.go](/e:/work/go/ctp-future-kline/internal/web/server.go)。

**4. 核心主链路：实时行情从 CTP 到落库**
这一章回答：tick 到底怎么从交易所一路走到文件和 MySQL。

主链路顺序是：

1. `CTP -> mdSpi.OnRtnDepthMarketData`
2. `mdSpi -> marketDataRuntime.onLiveTick`
3. `runtime -> shard`
4. shard 旁路把原始 tick 异步送到 `fileWriter`
5. shard 内完成时间规整、去重、漂移检测、1m 聚合
6. 封口后的 1m bar 继续生成 mm 和 L9
7. `persistTask -> dbBatchWriter -> klineStore -> MySQL`
8. 运行状态进入 `RuntimeStatusCenter`，再经 WebSocket/HTTP 暴露出去

这条链路里的关键设计思想是：

- 同一合约按 shard 串行，保证顺序
- 文件落盘和 DB 落盘都异步，避免阻塞 CTP 回调
- 1m 是源头，mm/L9 都建立在已封口 bar 之上
- 持久化层通过批量 upsert 提高吞吐
- 状态中心把延迟、队列、连接状态统一暴露

最容易误解的点有两个。第一，tick 文件落盘只是旁路记录，不决定业务聚合；第二，L9 不是单独行情源，而是根据同品种合约 1m 异步算出来的。

建议先看 [internal/quotes/md_spi.go](/e:/work/go/ctp-future-kline/internal/quotes/md_spi.go)、[internal/quotes/market_data_runtime.go](/e:/work/go/ctp-future-kline/internal/quotes/market_data_runtime.go)、[internal/quotes/market_data_db_writer.go](/e:/work/go/ctp-future-kline/internal/quotes/market_data_db_writer.go)。

**5. 核心辅链路：回放、导入、查询、策略、交易**
这一章回答：除了实时主链路，其他链路分别怎么挂上来。

回放链路的作用是“把历史事件重新送回实时链路”。`replay.Service` 负责调度，`ReplaySink` 负责把事件还原成 `tickEvent` 再交给 `mdSpi`。所以 replay 不是一套平行实现，而是复用实时处理逻辑。

导入链路的作用是“补齐历史数据”。`importer` 负责读取通达信数据、做冲突决策、写入 MySQL。它解决的是“实时系统上线前后的历史连续性”问题。

查询链路的作用是“把已经存好的 K 线和索引给前端图表用”。`klinequery` 面向图表 bars，`searchindex` 面向搜索和定位，两者一起支撑前端交互。

策略链路和交易链路是叠加层。`strategy` 通过 Python gRPC 服务承接策略实例、信号、回测；`trade` 继续承接账户、持仓、订单、成交和风控相关能力。它们依赖行情，但不属于行情底层本体。

最容易误解的点是：回放和导入都跟“历史”有关，但回放解决“复现运行时行为”，导入解决“补齐存储中的历史数据”。

建议先看 [internal/replay](/e:/work/go/ctp-future-kline/internal/replay)、[internal/importer](/e:/work/go/ctp-future-kline/internal/importer)、[internal/strategy](/e:/work/go/ctp-future-kline/internal/strategy)。

**6. 存储模型与关键数据表**
这一章回答：系统把什么数据存到了哪里。

市场数据表的主干是四类：

- `future_kline_instrument_1m_<variety>`
- `future_kline_instrument_mm_<variety>`
- `future_kline_l9_1m_<variety>`
- `future_kline_l9_mm_<variety>`

此外还有几类配套表：

- 图表布局与绘图：`chart_layouts`、`chart_drawings`
- 检索索引：`kline_search_index`
- 交易日历与会话：交易日历、交易时段相关表
- 用户配置与运行模式：`user_config`
- 交易相关：账户、订单、成交、审计等表

这里的核心建模思想是：市场数据按“普通合约/L9”和“1m/mm”分层，查询和展示时不必在单表里混合多种语义。

最容易误解的点是：`AdjustedTime` 和 `DataTime` 分别承担“连续交易时间轴”和“业务标签时间”职责，尤其夜盘场景下不能混看。

建议先看 [internal/quotes/kline_store.go](/e:/work/go/ctp-future-kline/internal/quotes/kline_store.go) 和 [internal/db](/e:/work/go/ctp-future-kline/internal/db)。

**7. Web/API 与前后端协作方式**
这一章回答：前端怎么驱动后端，后端怎么把状态推给前端。

`internal/web/server.go` 是整个控制面的核心入口。它负责：

- 提供 HTTP API
- 提供 WebSocket 状态推送
- 承载前端静态资源
- 把 app mode、replay、strategy、trade、calendar、importer 等子系统统一挂到一个控制台下

前端在 `web` 目录，是 Vue3 + Vite 工程。它不是简单展示页，而是运维控制台：可以启动服务、切换模式、触发回放、查看状态、查询图表、操作策略和交易功能。

最容易误解的点是：状态页上的很多指标不是“系统总历史统计”，而是“运行态快照或窗口指标”，用来做实时监控和排障。

建议先看 [internal/web/server.go](/e:/work/go/ctp-future-kline/internal/web/server.go) 和 [web/src/App.vue](/e:/work/go/ctp-future-kline/web/src/App.vue)。

- `db`
- `web`
- `calendar`
- `strategy`

运行环境上，这个工程有明显的 Windows/CTP 现实约束：

- 需要 CTP DLL
- 需要 MySQL
- 前端构建需要 Node
- 策略模块启用时需要 Python

配置设计的重点不是参数多，而是把实时链路、回放链路、交易链路、前端行为、队列容量、数据库 DSN 统一放在同一份配置中管理。

最容易误解的点是：某些 DSN 和逻辑库是运行时拼装出来的，不是所有字段都直接从 JSON 原样拿去用。

建议先看 [internal/config/config.go](/e:/work/go/ctp-future-kline/internal/config/config.go)。

**10. 推荐阅读路径与二次开发切入点**
这一章回答：新人第一天该怎么读，后续从哪里切入开发最稳。

推荐阅读顺序：

1. 先读 [README.md](/e:/work/go/ctp-future-kline/README.md)，建立功能地图
2. 再读 [internal/config/config.go](/e:/work/go/ctp-future-kline/internal/config/config.go)，理解运行参数和模式
3. 然后读 [internal/quotes](/e:/work/go/ctp-future-kline/internal/quotes)，只抓主链路
4. 再读 [internal/replay](/e:/work/go/ctp-future-kline/internal/replay)，理解“历史如何复现为实时行为”
5. 接着看 [internal/web/server.go](/e:/work/go/ctp-future-kline/internal/web/server.go)，理解控制面
6. 最后按需要进入 `importer`、`klinequery`、`strategy`、`trade`

二次开发切入点通常有四类：

- 行情处理与聚合：进 `quotes`
- 图表查询与检索：进 `klinequery/searchindex`
- 控制台与 API：进 `web`
- 策略或交易扩展：进 `strategy/trade`

最重要的建议是：不要一开始就按目录平铺看代码，先围绕一条主链路建立因果关系，再扩展到其他模块。这样上手速度会快很多。
