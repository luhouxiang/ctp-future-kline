# ctp-future-kline 源码上手指南

这份文档面向准备接手本项目源码的开发者，目标不是穷举所有文件和 API，而是帮助读者先建立整体结构，再快速进入关键链路和关键模块。

文档采用“先总纲，再分目细化”的方式组织。建议先完整读一遍总纲，再按自己的工作方向深入相应章节。

## 总纲

这个工程是一个以国内期货 CTP 为实时行情入口、以 MySQL 为核心存储、以 Web 控制台为操作面的综合型行情平台。

它不只是“采集 1 分钟线”的工具，而是一套把实时行情、历史导入、回放复现、图表查询、策略接入、模拟交易、运行状态监控整合在一起的系统。

从系统能力上看，它主要解决三类问题：

- 把交易所实时行情可靠地接进来，并沉淀成结构化数据
- 把历史数据、实时数据和图表查询统一到同一套数据底座上
- 在统一行情底座上继续叠加回放、策略、交易和控制台能力

从子系统划分上看，可以把它理解为 7 个核心部分：

- `quotes`：实时行情主链路
- `replay`：历史回放调度与复现
- `importer`：历史分钟线导入
- `klinequery/searchindex`：图表数据查询和检索
- `web`：HTTP API、WebSocket、前端控制台
- `strategy`：Python gRPC 策略服务与回测
- `trade`：交易服务与账户/订单/成交管理

新人最先要建立的心智模型是：

`quotes -> replay/importer/klinequery/web -> strategy/trade`

也就是先理解实时行情主链路，再理解历史、查询和控制面，最后再进入策略和交易扩展层。

## 1. 工程定位与核心能力

### 这一章回答什么问题

这一章回答：这个工程本质上是什么系统，它的能力边界在哪里。

### 这个模块在系统里的位置

项目的定位不是单一“行情采集器”，而是“期货行情与图表/策略平台的内核”。

它从 CTP 实时接收 tick，生成普通合约和 L9 的 1m / mm K 线，落到 MySQL，再通过 Web 控制台、查询接口、回放能力、策略服务和交易服务把这些数据继续利用起来。

### 关键入口 / 核心对象 / 核心数据

从高层看，核心对象不是某个 struct，而是三类系统能力：

- 实时能力：CTP tick -> runtime -> K 线
- 存储能力：tick CSV / MySQL / 查询索引
- 控制能力：Web、回放、策略、交易、配置、状态监控

### 它和上下游如何交互

上游是：

- CTP 行情前置
- 通达信历史数据文件
- tick CSV / bus 回放源

下游是：

- MySQL 数据库
- 前端图表和控制台
- Python 策略服务
- 模拟或实盘交易模块

### 新人最容易误解的点

- 这个工程不是一个“只负责采样写库”的单功能程序
- 它也不是策略系统本体，而是策略和交易所依赖的数据与控制平台
- `quotes` 虽然只是一个目录名，但它其实是整个工程的核心域

## 2. 运行形态与系统边界

### 这一章回答什么问题

这一章回答：系统有哪些运行模式，这些模式之间如何分界。

### 这个模块在系统里的位置

这个工程既支持实时运行，也支持回放运行；既支持模拟交易，也支持真实交易。理解这些边界，是读懂 Web 控制台、交易模块和 replay 模块的前提。

### 关键入口 / 核心对象 / 核心数据

最重要的两组边界是：

- 行情来源边界：`live` 与 `replay`
- 交易后端边界：`live_trade`、`paper_live`、`paper_replay`

数据侧还有逻辑数据库边界：

- 共享元数据
- 实时市场数据
- 回放市场数据
- 图表用户数据
- 交易数据

### 它和上下游如何交互

实时模式下，行情直接来自 CTP，写入实时市场库。

回放模式下，历史 tick 从 tick CSV 或 bus 中读出，再重新送回实时处理链路，写入回放市场库，并驱动模拟交易和图表。

### 新人最容易误解的点

- replay 不是另写一套行情聚合逻辑，而是尽量复用实时链路
- paper/live 的区别不在图表，而在交易后端和数据模式
- 共享元数据库不是市场行情库，它承载的是跨模式共用信息

## 4. 核心主链路：实时行情从 CTP 到落库

### 这一章回答什么问题

这一章回答：tick 到底如何从交易所一路走到 tick 文件和 MySQL。

### 这个模块在系统里的位置

这是全工程最重要的一章。理解这条主链路，等于理解了工程的核心运行方式。

### 关键入口 / 核心对象 / 核心数据

主链路可以概括为：

`CTP 回调 -> mdSpi -> marketDataRuntime -> shard -> fileWriter / dbWriter -> klineStore -> status/web`

其中最关键的对象是：

- `mdSpi`
- `marketDataRuntime`
- `marketDataShard`
- `dbBatchWriter`
- `klineStore`
- `RuntimeStatusCenter`

### 它和上下游如何交互

主流程如下：

1. CTP 回调进入 `OnRtnDepthMarketData`
2. `mdSpi` 提取并清洗 tick 字段，封装成内部 `tickInputData`
3. `marketDataRuntime` 把 tick 按合约 hash 到 shard
4. 同时旁路异步写 tick CSV
5. shard 内完成时间规整、去重、漂移检测和 1m 聚合
6. 1m bar 封口后继续触发 mm 和 L9 计算
7. 持久化任务进入 `dbBatchWriter` 批量写库
8. 状态与延迟指标进入 `RuntimeStatusCenter`
9. WebSocket / HTTP 对外暴露运行状态

### 新人最容易误解的点

- tick 文件落盘只是旁路记录，不决定 K 线计算逻辑
- 1m 是源头，mm 和 L9 都建立在已经封口的 bar 之上
- DB writer 是异步批量写库，不是“来一条写一条”
- 状态页看到的是运行态指标，不是离线统计报表

## 5. 核心辅链路：回放、导入、查询、策略、交易

### 这一章回答什么问题

这一章回答：除了实时主链路以外，其他重要链路各自是干什么的。

### 这个模块在系统里的位置

这些链路都不是独立世界，而是围绕实时数据底座展开。

### 关键入口 / 核心对象 / 核心数据

回放链路：

- `replay.Service`
- `ReplaySink`

导入链路：

- `importer` 各类 session / conflict decision

查询链路：

- `klinequery`
- `searchindex`

策略链路：

- `strategy.Manager`
- Python gRPC 服务

交易链路：

- `trade.Service`

### 它和上下游如何交互

回放链路负责把 tick CSV 或 bus 事件重新还原成 `tickEvent`，再交回 `quotes` 主链路，因此 replay 的价值在于复现线上行为。

导入链路负责把通达信历史分钟线写入 MySQL，补齐“实时链路接入之前”的历史区间。

查询链路从 MySQL 和索引表中取数，服务前端图表与检索。

策略和交易链路依赖市场数据和控制面，但自身不生产底层行情数据。

### 新人最容易误解的点

- replay 和 importer 都和“历史”有关，但目的完全不同
- strategy 和 trade 是叠加层，不是行情主链路的一部分
- klinequery 面向图表查询，searchindex 面向快速检索

- [internal/replay/service.go](../../internal/replay/service.go)
- [internal/importer](../../internal/importer)

## 6. 存储模型与关键数据表

### 这一章回答什么问题

这一章回答：系统到底把什么数据存到了哪里。

### 这个模块在系统里的位置

MySQL 是工程的核心持久化层。图表、查询、回放、策略和运维都间接依赖这里的存储模型。

### 关键入口 / 核心对象 / 核心数据

市场数据表的主干是四类：

- `future_kline_instrument_1m_<variety>`
- `future_kline_instrument_mm_<variety>`
- `future_kline_l9_1m_<variety>`
- `future_kline_l9_mm_<variety>`

除此之外还有：

- 图表布局和绘图对象表
- K 线搜索索引表
- 交易日历和交易时段相关表
- 用户配置表
- 交易相关表

### 它和上下游如何交互

实时和导入链路往这些表里写数据，查询和前端图表从这些表里取数据，搜索依赖索引表做加速。

### 新人最容易误解的点

- `DataTime` 与 `AdjustedTime` 不是一个概念，夜盘尤其要分清
- L9 表不是交易所原生数据，而是系统异步计算的结果
- mm 表不是通过 SQL 再聚合出来，而是由实时链路直接生成

- [internal/quotes/kline_store.go](../../internal/quotes/kline_store.go)

## 7. Web/API 与前后端协作方式

### 这一章回答什么问题

这一章回答：前端如何驱动后端，后端如何把运行态反馈给前端。

### 这个模块在系统里的位置

`web` 子系统是整个工程的控制面。它不是可有可无的展示页，而是运维、观察和操作的统一入口。

### 关键入口 / 核心对象 / 核心数据

关键入口是：

- `internal/web/server.go`
- `web/src/App.vue`

对外方式有两类：

- HTTP API：执行操作、获取数据
- WebSocket：实时推送状态和事件

### 它和上下游如何交互

前端通过 HTTP 调用启动服务、发起回放、导入历史、查询 bars、切换模式；后端通过 WebSocket 持续推送状态、回放进度、策略和交易更新。

### 新人最容易误解的点

- 状态页不是静态信息汇总，而是系统运行中的实时快照
- 前端很多操作不是“查询”，而是直接驱动后端状态机
- app mode 会影响行情来源、图表数据模式和交易后端

### 建议先看哪些文件

### 这一章回答什么问题

### 关键入口 / 核心对象 / 核心数据

### 它和上下游如何交互

当线上出现异常时，通常会这样排查：

## 结语

如果只允许带着一个问题去进入这个工程，建议带着这个问题：
